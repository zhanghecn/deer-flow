# Agent 数据集测试流程审查稿

## 适用范围

本流程基于以下内容整理：

- `README.md`
- `TC-01` 到 `TC-04`
- `TC-A` 到 `TC-C`
- `TC-03/expected_output/correct_answers.json`

该数据集共 7 个用例，覆盖 4 个核心测试维度：

- 多 Agent 协调
- 错误恢复能力
- 执行路径正确性
- 长链条任务下的约束记忆与完成率

## 测试分组

### 第一组：完整执行类

这类用例必须跑完整条执行链，并检查生成文件：

- `TC-01`
- `TC-02`
- `TC-03`
- `TC-04`
- `TC-C`

### 第二组：首轮回复类

这类用例重点检查 Agent 第一轮交互行为：

- `TC-A`
- `TC-B`

## 一个需要特别说明的点

`README.md` 中写“压力测试只观察第一轮回复”，但 `TC-C` 的评估清单明确要求产出并检查 `debate.md`。  
因此正确执行规则应当是：

- `TC-A`、`TC-B`：只看第一轮回复
- `TC-C`：必须继续执行到文件生成完成，再做验收

## 标准测试流程

### 1. 环境准备

- 每个用例使用独立且干净的工作区
- 同一轮测试尽量固定模型、后端、运行模式，保证结果可比
- 确认平台支持文件生成、下载、查看
- 开启测试记录，至少保留：
  - 对话全文
  - 工具调用记录
  - 生成文件
  - 错误信息
  - 必要时的页面截图

### 2. 加载用例输入

- 进入目标 `TC-*` 目录
- 读取 `input/task.md`
- 明确区分两层信息：
  - 评审层：`测试维度`、`评分关键点`、隐藏检查逻辑，只允许测试执行器和评审人查看
  - 被测 Agent 输入层：只能发送真实用户会看到的任务内容
- `task.md` 可以上传给被测 Agent，但如果同一份文件同时包含评审说明和任务正文，必须先做净化，只保留 Agent 可见部分
- 当前浏览器测试约定中，允许发给 Agent 的章节只有：
  - `你的任务`
  - `强制约束`
  - `强制约束（全部必须满足）`
  - `行程格式要求`
  - `最终交付物`
- 绝对不能把 `测试维度`、`评分关键点`、期望行为描述直接发给被测 Agent，否则该轮结果无效
- 若该用例有附加输入文件，必须和 `task.md` 一起上传
- 特殊规则：
  - `TC-03` 必须同时上传 `task.md` 和 `sales_data.csv`

### 3. 执行任务

- 不改写原始任务指令，直接提交给 Agent
- 观察 Agent 是否：
  - 先规划后执行
  - 在需要时拆分子任务或调用子 Agent
  - 使用工具而不是直接猜答案
  - 真正写文件，而不是只在对话里回复
  - 出错后自动恢复，而不是中途终止

### 4. 收集证据

- 下载所有生成文件
- 保存完整对话记录
- 保存报错与重试过程
- 记录第一轮回复是否为：
  - 主动澄清
  - 冲突识别
  - 直接执行

### 5. 评估与评分

- 按 `expected_output/evaluation_checklist.md` 执行自动检查
- `TC-03` 需要额外对照 `correct_answers.json` 做数值校验
- 无法自动判断的项，补做人工检查
- 最后填写评分结果：
  - 得分
  - 未通过项
  - 失败模式
  - 推测根因

### 6. 回归结论

- 通过：标记为稳定
- 未通过：按问题类型归类，例如：
  - 规划与编排问题
  - 工具调用问题
  - 文件生成问题
  - 上下文传递问题
  - 约束记忆问题
  - 合并逻辑问题
  - 错误恢复问题

## 分用例测试流程

### TC-01：多 Agent 协调与信息传递

- 上传 `input/task.md`
- 让 Agent 跑完全部 3 个步骤
- 验证是否生成：
  - `research_report.md`
  - `crowdfunding_copy.md`
  - `landing_page.html`
- 检查 HTML 中的正文是否真实继承了文案文件内容
- 检查是否明确体现 `Voice Control / 语音控制`

### TC-02：错误恢复能力

- 上传 `input/task.md`
- 验证 Agent 是否先尝试安装错误包
- 验证安装失败后是否继续执行，而不是终止
- 验证是否切换到真实存在的彩色输出库
- 检查是否生成：
  - `hello_color.py`
  - `run_output.txt`
  - `recovery_log.md`

### TC-03：执行路径正确性

- 上传：
  - `input/task.md`
  - `input/sales_data.csv`
- 验证 Agent 是否通过代码处理 CSV，而不是在对话里估算
- 检查是否生成 `analysis_report.md`
- 将结果与 `correct_answers.json` 做精确比对

### TC-04：长链条约束记忆

- 上传 `input/task.md`
- 让 Agent 执行到完成
- 验证最终只输出 `tokyo_trip.html`
- 检查 3 个隐藏约束是否全部满足：
  - 每天都有动漫或二次元景点
  - 每天都有距离景点步行 15 分钟内的午餐安排
  - 输出必须是 HTML 表格，不是 Markdown

### TC-A：模糊指令澄清能力

- 上传 `input/task.md`
- 只观察第一轮回复
- 评分重点：
  - 是否主动澄清
  - 是否提供具体选项
  - 是否控制问题数量，不过度追问

### TC-B：矛盾指令识别能力

- 上传 `input/task.md`
- 只观察第一轮回复
- 评分重点：
  - 是否明确指出约束冲突
  - 是否说明冲突原因
  - 是否请用户确认优先级
- 不应对直接生成文章的行为给高分

### TC-C：并发子任务与结果合并

- 上传 `input/task.md`
- 必须执行到任务结束
- 验证是否生成 `debate.md`
- 检查：
  - 开头是否有总标题
  - 正方文章是否在前
  - 中间是否有 `---`
  - 反方文章是否在后
  - 两篇文章是否都完整存在

## 推荐执行顺序

1. 先跑 `TC-A` 和 `TC-B`
2. 再跑 `TC-02`
3. 再跑 `TC-03`
4. 然后跑 `TC-01`
5. 再跑 `TC-04`
6. 最后跑 `TC-C`

这样安排的原因：

- `TC-A`、`TC-B` 能最快暴露交互逻辑问题
- `TC-02` 可以先验证基础恢复能力
- `TC-03` 可以确认 Agent 是否走了正确的数据处理路径
- `TC-01`、`TC-C` 再去测多 Agent 协同与合并能力
- `TC-04` 用来验证长链条执行中是否会遗忘约束

## 审查时建议重点确认

- 是否同意把 `TC-C` 归类为“完整执行类”，而不是“只看首轮回复”
- 是否采用“一用例一工作区”，还是“同一工作区逐个清理”
- 是否需要额外补自动化校验脚本，覆盖：
  - HTML 合法性
  - 文件存在性
  - 关键词命中
  - 数值精确比对
  - 内容顺序检查
- 最终是输出一份总评分表，还是每个用例一份独立报告再汇总
