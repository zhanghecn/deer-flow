## Internal Evidence Findings

From admin trace inspection:

- Attached-document turns currently bypassed the legacy knowledge-document listing tool.
- The agent worked directly from attached KB context into `get_document_tree` and `get_document_evidence`.
- For the annual report figure question, `get_document_evidence` returned `visual_summary`, but `evidence_blocks` only contained `kind=text`.
- In the audited annual-report evidence calls:
  - `image_path` was empty
  - `image_markdown` was empty
- Because of that, the agent had no concrete image asset to inline, only text plus visual summaries.

This is the key reason the figure-oriented answer did not naturally render an image in chat.

## Concrete Problems Confirmed

1. Knowledge management page refresh is stale after upload.
   - Backend had already built the markdown KB successfully.
   - Filtered management UI still showed `0 documents` until reload.

2. Image-grounded answers are not truly image-grounded yet.
   - Visual summaries exist.
   - Actual evidence blocks did not carry usable image asset references.
   - No inline image appeared in chat.
   - No `get_document_image` call happened in the audited figure question.

3. Figure-oriented retrieval is too expensive.
   - The annual report question consumed `778394` total tokens.
   - The trace explored several irrelevant subtrees before settling on text/table evidence.

4. Attached-doc turns do not currently show the ideal minimal tool path.
   - In both audited traces, the model did not call the legacy knowledge-document listing tool.
   - That is not necessarily wrong for attached-doc turns, but it is the actual behavior of the current stack and should be kept in mind when evaluating prompt/tool policy.

## Recommended Next Focus

1. Fix management-page refresh / query invalidation after KB creation.
2. Ensure image-bearing evidence blocks carry real asset refs:

3. Add prompt/tool policy so figure-dependent answers prefer visual evidence, not only textual summaries.
