# Testing Checklist

Use this checklist when validating a real customer integration or preparing handoff.

## Pre-checks

1. Confirm the target agent is published.
2. Confirm the bearer key belongs to the published public API surface.
3. Confirm the caller is using the correct gateway base URL for the target environment.

## Functional checks

1. A blocking call returns a valid turn snapshot and non-empty `output_text`.
2. A streaming call emits `turn.started` and ends in either `turn.completed`, `turn.requires_input`, or `turn.failed`.
3. When thinking is enabled, `assistant.reasoning.delta` appears and the UI accumulates it into one reasoning area.
4. If the agent uses tools, the UI shows:
   - tool method name
   - `tool_arguments`
   - `tool_output`
5. A follow-up request with `previous_turn_id` continues the same thread.
6. If artifacts are returned, the UI or backend stores the final `download_url`.
7. If the project is a browser demo generated by `lead_agent`, the final preview entry is under `/mnt/user-data/outputs/<project-name>/index.html`.
8. The generated frontend does not reference missing sibling files such as `styles.css`.

## Failure checks

1. HTTP non-2xx request errors are visible to the user.
2. `turn.failed` is visible to the user and does not silently discard partial output.
3. `turn.requires_input` is rendered as waiting, not completed.
4. Opening the generated preview from the workspace does not return `404 artifact not found`.

## Handoff checklist

Before handing the integration to a customer team, make sure they have:

- exact `base_url`
- exact published `agent` name
- exact auth method
- one working blocking example
- one working streaming example
- one follow-up turn example with `previous_turn_id`
- explicit guidance for rendering reasoning and tool-call steps
